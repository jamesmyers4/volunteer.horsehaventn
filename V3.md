V3.md — Horse Haven of Tennessee Ops Platform — V3 Buildout Sessions

How to use this document
Same discipline as V2.md. Seven sessions, each its own Claude Code session. At the end of every session:

The feature described in that session must be fully implemented.
A matching Playwright/TypeScript test suite must be written or extended, following existing test conventions already in this repo.
Claude Code must STOP and wait for the user to start a new session before moving to the next one, even if this file is loaded in full.
Before starting any session, read the current prisma/schema.prisma, CONTEXT.md, and CLAUDE.md. This document describes intent and field-level requirements, not literal Prisma syntax — resolve naming to match whatever conventions those files already establish (cuid IDs, no hard deletes on business entities, camelCase, admin-editable lookup-table pattern, ChangeLog auto-capture via the Prisma Client Extension, append-only where noted, baseline+override for feeding, computed-not-cached for anything derivable). If a described model conflicts with something that already exists, extend it rather than duplicating it, and note the deviation in the session summary.

At the end of each session, create a brief summary and status (Session ## complete).

Terminology and roles are unchanged: core entity is Animal, roles are ADMIN, SHIFT_LEAD, VOLUNTEER, GUEST. Session 4 below adds occurrence-scoped shift leadership (Shift.assignedLeadId) on top of the existing global role system — it does not redesign roles/permissions.

Source data: July 19, 2026 site visit
The user photographed the live Horse Care Board (Google Sheets) and the physical barn/pasture layout during a volunteer shift. Confirmed facts, grounding the sessions below:

Feeding is tracked as five distinct fields per animal, separately for AM and PM: Grain, Hay, Skin Care, Meds, Handling Notes. The same animal's AM and PM entries routinely differ.
Meds/skin-care instructions frequently carry an explicit end date in free text ("End 7/24/26", "x5 days, end 7/20").
Stalls/pastures are grouped (M1–M4, L1–L6, RP1, RP6); the barn also has non-stall functional spaces (Wash Bay, Feed Room, Scale, Tools/Carts, Bathrooms, Kitchen) — consistent with CONTEXT.md §10's existing rule that non-horse facility markers aren't DB-backed, just map annotations.
Trough cleaning and horse weighing rotate on a fixed weekly pattern by pasture group. Stall cleaning also rotates by weekday in ranges.
Stalls aren't uniform: most get a regular clean (scoop, sift, refresh); a small subset ("strip stalls," e.g. Remus's and Rowan's) get a full strip-out each time — this is a property of the stall, not a one-off task choice. (Matches CONTEXT.md §17's existing note that strip stalls get a chlorhexidine wash + fresh shavings.)
An adoption/status board tracks Currently in Foster, Pending Adoption, and Adopted in Last 90 Days, each row showing "HHT Days" (days at Horse Haven) computed from intake.
Animals are frequently intake in batches of 2–14, usually sharing a naming theme (e.g. an Irish-themed group: Rowan, Shannon, Fiona, Guinness, Sorcha, Bridgette; a luxury-brand group: Gucci, Prada, Vuitton, Armani). Members of a themed group are marked with an asterisk on the board.

Two, occasionally three, animals from the same intake group are often adopted out together, since bonded animals are kept together when possible.
The asterisk's exact meaning is unconfirmed by the user — do not model it.

Session 1 — Intake Groups, Lineage & Adoption Status
Goal: Track animals that arrived together as a cohort, optionally record real family relationships between individual animals (sparse — most will have none), extend the existing status/placement tracking to match the real Foster/Pending Adoption board states, and support two or more animals being adopted out together.

Data model:

IntakeGroup: id, label (often thematic, e.g. "Irish Group"), intakeDate, notes (free text), isActive. New model — not ChangeLog-tracked (lookup-ish, same category as EventCategory).
Animal.intakeGroupId (nullable FK to IntakeGroup). New field. Most animals will have none — this is optional, not required.
AnimalRelationship: id, animalId, relatedAnimalId, relationType enum (SIRE_OF, DAM_OF, SIBLING_OF, OTHER), notes (nullable), plus proper Volunteer FK fields for who recorded it (recordedById / recordedBy relation), matching the AnimalLocationAssignment/EventSignup convention rather than the older plain-string performedBy/recordedBy pattern seen on CareEntry/MedicationLog/WeightEntry/AnimalMetric. Store one row per real-world relationship — do not also store the inverse row; derive the inverse at display/query time (a SIRE_OF row with animalId = Guinness and relatedAnimalId = Rowan should render as both "Guinness's offspring: Rowan" and "Rowan's sire: Guinness"). New model. Recommend ChangeLog tracking it, since it touches the same legal-defensibility surface as Animal itself (§4/§11) for legalCase animals — confirm with the user if that seems like overkill.
AnimalStatus enum: add FOSTER and PENDING_ADOPTION as new values alongside the existing ACTIVE, ADOPTED, RETURNED, DECEASED, TRANSFERRED. No new status model needed — Animal.status already exists.
Placement.placementGroupId (nullable String, a shared token rather than a new table) — set to the same value across two or more Placement rows created for the same real-world adoption event, so co-adopted animals can be queried as a set. Deliberately simple: no new PlacementGroup table, since Placement already carries adopterName/adopterContact/placedDate and nothing else needs to hang off the "group" concept beyond linking rows together.

Business rules:

IntakeGroup assignment and AnimalRelationship edits are restricted to ADMIN/SHIFT_LEAD, matching the Location-assignment permission pattern from V2 Session 1.
An animal's detail page should surface its IntakeGroup (with a link to the rest of the group) and any AnimalRelationship rows involving it in both directions, if any exist.
HHT Days = Placement.placedDate − Animal.intakeDate if a Placement with placedDate exists, otherwise today − Animal.intakeDate. Compute at query time — Animal.intakeDate already exists, don't add a duplicate field.
Moving Animal.status to ADOPTED should coincide with a Placement row being created/finalized (placedDate set) — confirm the existing status-change flow already enforces or assumes this pairing before adding new logic on top of it.
No hard deletes — deactivate an IntakeGroup via isActive.

Test coverage to add:

IntakeGroup CRUD; Animal.intakeGroupId remains nullable and unenforced.
AnimalRelationship creation and inverse-direction display — specifically test the real scenario: Guinness SIRE_OF Rowan and Guinness SIRE_OF Shannon, with Bridgette DAM_OF Rowan and Sorcha DAM_OF Shannon, confirming an animal can carry both a sire and a dam relationship independently.
FOSTER and PENDING_ADOPTION status transitions behave like existing AnimalStatus values everywhere status is already used (filters, dashboards).
Two Placement rows sharing a placementGroupId are queryable together; HHT Days computed correctly pre- and post-placement.
Permission checks: only ADMIN/SHIFT_LEAD can edit groups/relationships.

Session 2 — Recurring Facility Tasks (Trough & Stall Care)
Goal: Model weekly-rotating facility upkeep (trough cleaning, regular stall cleaning, strip-stall cleaning) so it can be scheduled by an admin via a monthly calendar, logged via a simpler quick-add path outside the calendar, and checked off by whoever's on shift.

Data model:

FacilityTaskType (admin-editable lookup, same pattern as WorkType/FeedType/CareType — not ChangeLog-tracked): seed TROUGH_CLEAN, STALL_CLEAN, STALL_STRIP.
Location.requiresStripClean (boolean, default false) — new field on the existing Location model, BARN_STALL-specific (true for Remus's and Rowan's stalls today), nullable/unused for other types, same pattern already used for barnNumber/stallNumber.
RecurringTaskTemplate: id, taskTypeId, targetLocationId (FK to Location — a FIELD-type location for TROUGH_CLEAN, a BARN_STALL-type location for STALL_CLEAN/STALL_STRIP; validate the pairing app-side at the Server Action layer, same convention already used for Location's own per-type field validation, not a DB constraint), dayOfWeek (int, 0–6), shiftType (reuse the existing ShiftType enum, AM/PM), isActive. This is what the admin calendar manages.
FacilityTaskCompletion: id, taskTypeId, targetLocationId, date, shiftType, completedById (Volunteer FK, following the AnimalLocationAssignment/EventSignup FK convention), notes (nullable), templateId (nullable FK to RecurringTaskTemplate — null means logged ad hoc, not off a recurring pattern).

Business rules:

Do not pre-generate a row per expected occurrence. A given day/shift's expected task list is derived at render time by matching RecurringTaskTemplate rows against today's dayOfWeek + shiftType — the same derive-don't-cache approach already used for current location and (per the schema comments) tier. A FacilityTaskCompletion row only gets created when someone actually checks a task off.
The quick-add path lets a volunteer log a FacilityTaskCompletion directly (taskType + location + date/shift + optional notes) with no templateId, for anything done outside the recurring pattern.
Only ADMIN/SHIFT_LEAD can create/edit RecurringTaskTemplate rows. Any signed-in volunteer on a shift can log a FacilityTaskCompletion for that shift.
STALL_STRIP should only be assignable/loggable against Locations where requiresStripClean is true.
No hard deletes — deactivate a RecurringTaskTemplate via isActive.

Test coverage to add:

RecurringTaskTemplate CRUD; weekday/shift expansion returns the correct task list for a given date.
Location.requiresStripClean correctly routes a stall into STALL_STRIP vs STALL_CLEAN.
Ad hoc FacilityTaskCompletion creation without a templateId.
Completing the same recurring task twice on the same day/shift doesn't produce a confusing duplicate "still pending" state.
Permission checks: only ADMIN/SHIFT_LEAD manage templates; any active volunteer can log a completion.

Session 3 — Shift Alerts & Announcements
Goal: Let admins broadcast time-relevant information (weather closures, new rules, seasonal reminders) that surfaces as a pinned banner and gives shift leaders something concrete to brief. Extend the existing ChatChannel/ChatMessage system (BROADCAST and SHIFT channel types, per CONTEXT.md §14) rather than building a parallel notification model.

Data model:

ChatChannel.shiftType (nullable, reuse existing ShiftType enum) — set only when type is SHIFT, so a shift-scoped channel is queryable directly instead of matched by name string, mirroring how ShiftTemplate.shiftType already works.
ChatMessage.severity (new enum AlertSeverity: INFO, WARNING, URGENT — nullable; ordinary chat messages leave this null).
ChatMessage.expiresAt (nullable DateTime — some alerts are standing reminders with no expiry, e.g. a cold-weather reminder; most regular chat messages won't set this).
ChatMessage.pinned (boolean, default false).

Business rules:

Only ADMIN can create a message with pinned = true. A non-admin attempting to set it should be rejected server-side, not just hidden in the UI — regular chat posting permissions elsewhere are unchanged.
A message is a "live alert" when pinned is true and now falls between createdAt and expiresAt (or expiresAt is null). Live alerts in the BROADCAST channel render as a banner across every authenticated view; live alerts in a SHIFT-type channel render only for volunteers viewing/on that matching shiftType.
severity drives banner styling only, not gating logic.
Pinned status doesn't remove a message from normal chat history — it's the same message, just additionally surfaced. No hard delete needed for expiry; expiresAt passing simply stops the banner from rendering.

Test coverage to add:

Pinning permission: only ADMIN can set pinned = true.
Live-alert window filtering at the createdAt/expiresAt boundaries, and standing alerts (null expiresAt) never auto-expire.
Scope filtering: a pinned message in a SHIFT-type channel with shiftType = AM only banners for AM-shift context; BROADCAST pins banner everywhere.
Expired or unpinned messages don't render as a banner but remain visible in normal chat history.

Session 4 — Shift Roster & Leader-Driven Attendance
Goal: Let the volunteer leading a specific shift occurrence manage attendance for that shift as a bulk roster action — default everyone present, uncheck absentees — instead of requiring every volunteer to individually check in/out. The existing individual QR/kiosk flow stays fully intact as an alternate path.

Data model:

Shift.assignedLeadId (nullable FK to Volunteer) — new field on the existing Shift model. Occurrence-scoped: names who's leading this specific shift, layered on top of the existing global SHIFT_LEAD role, not a replacement for it. Shift is already ChangeLog-tracked per its existing schema comment, so this inherits that automatically.
No new tables. Attendance itself continues to be represented by CheckIn, which already has everything this needs — including checkInMethod: ADMIN_ENTRY, which reads as though it was already anticipated for exactly this feature.

Business rules:

Default expected roster for a given Shift = Volunteers with an active RegularShiftAssignment (active = true, date within startDate/endDate) matching that Shift's dayOfWeek and shiftType, plus any Volunteer who already has a CheckIn row for that Shift (covers walk-ons/fill-ins who self-checked-in via QR/kiosk without a standing RegularShiftAssignment).
Bulk submit: for each volunteer marked present who does not already have a CheckIn row for this Shift, create one with checkInMethod = ADMIN_ENTRY, loggedById = the submitting leader's Volunteer id, checkInAt/checkOutAt defaulted from the resolved ShiftTemplate time (respecting FarmSettings.activeSeason and any Shift.actualStartTime/actualEndTime override), unless the leader enters different values directly. A volunteer who already has a CheckIn row for this Shift (e.g. self-checked via QR) is left untouched — their real method/times are never overwritten by the bulk action.
The leader can add any Volunteer to the bulk-submit list, not just the default roster — this covers fill-ins with no standing RegularShiftAssignment.
Confirm whether a volunteer can already edit their own CheckIn row's times after creation (e.g. to correct an ADMIN_ENTRY default) — if that path doesn't exist yet, this session needs to add it.
Only Shift.assignedLeadId, or a Volunteer holding global role ADMIN or SHIFT_LEAD, can run the bulk-submit action for a given Shift.

Test coverage to add:

Default roster derivation from RegularShiftAssignment for a given Shift's date/dayOfWeek/shiftType.
Bulk submit creates CheckIn rows only for volunteers without one yet — never duplicates or overwrites an existing self-check-in.
Adding a non-rostered walk-on volunteer to the bulk submit works.
Time defaults resolve correctly through ShiftTemplate + FarmSettings.activeSeason + any Shift-level actualStartTime/actualEndTime override.
Permission checks: only assignedLeadId / global ADMIN / global SHIFT_LEAD may submit.
A volunteer can edit their own ADMIN_ENTRY-created CheckIn row afterward (or this capability is added if missing).

Session 5 — End-of-Shift Checklist (Skeleton)

One ShiftReport per Shift.

Session 6 — Care Board Parity (Content & Display, Not Schema)
Goal: Confirm the existing feeding/care/medication models already capture what the real Horse Care Board tracks, and close the one real content gap. This session is mostly an audit — expect it to touch seed data and display logic more than prisma/schema.prisma.

Findings against the current schema:

FeedingBaseline.shift already splits Grain/Hay by AM/PM — no schema change needed.
MedicationRegimen.startDate/endDate already supports time-boxed meds ("end 7/24," "x5 days") — no schema change needed.
Skin Care items (fly masks, topical sprays) fit the existing CareType/CareEntry model — CONTEXT.md §12 already scopes CareType.category (SEASONAL/GROOMING) to cover exactly this. Check whether any CareType rows already exist for skin-care-style items; if not, seed some (e.g. "Fly Mask," "Topical Spray," "Leg Wrap"). This is a data task, not a schema task.
Gap: short-lived, date-specific "ATTN" handling flags seen on the real board (e.g. "cold hose if breaths per min exceed 20") don't cleanly fit Animal.handlingNotes, which is a single evergreen free-text field meant for standing guidance (paired with the RED-color soft warning per CONTEXT.md §6). Recommend modeling these as CareEntry rows instead — already dated, already free text, already categorized — rather than extending handlingNotes into something it wasn't designed for. Confirm this split with the user before implementing, since it changes how staff enter that data day to day.

Test coverage to add:

CareEntry correctly represents a skin-care instruction distinct from a MedicationRegimen entry for the same animal/day.
Whatever Feed Board query exists (or gets built) returns FeedingBaseline (grain/hay), CareEntry (skin care), MedicationRegimen (meds, respecting active date range), and Animal.handlingNotes as four distinct, correctly-attributed sources per animal per shift.
A MedicationRegimen with endDate in the past is excluded from "current" med displays but remains queryable via MedicationLog history.
