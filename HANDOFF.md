## V2 Rename Session — `Horse` → `Animal` (2026-07-18)

Before starting `V2.md` Session 1 (the Location model generalization), V2.md's own terminology note called for the core entity to be `Animal` rather than `Horse`, to cover mules, donkeys, minis, ponies, and non-equine animals — but the entire codebase up to this point was built around `Horse`. Rather than guess, this was raised with James directly: full rename now, as its own dedicated session separate from Location work, with a `species` enum added at the same time. Full detail on the reasoning is in `CONTEXT.md` §18; this is what actually happened.

- **Schema:** `Horse` → `Animal`, `HorseStatus` → `AnimalStatus`, `HorseSex` → `AnimalSex`, `HorsePhoto` → `AnimalPhoto`, `HorseMetric` → `AnimalMetric`, every `horseId` FK → `animalId` (including `DonationInKind.linkedHorseId` → `linkedAnimalId`). New `AnimalSpecies` enum (`HORSE | DONKEY | MULE | MINI_HORSE | PONY | CAT | OTHER`), new `Animal.species` field defaulting to `HORSE`.
- **Migration was hand-written**, not auto-generated — `prisma migrate dev`'s non-interactive diff warned it would drop the live `Horse` table (1 row in the local test DB, and real data in Neon), since it can't infer a rename without a prompt. The actual migration (`prisma/migrations/20260718223633_rename_horse_to_animal/migration.sql`) uses `ALTER TABLE/COLUMN/TYPE ... RENAME` throughout — verified with `prisma migrate diff --from-config-datasource --to-schema` afterward, which reported zero drift, before it was ever pointed at production.
- **Binx backfill:** queried the live Neon DB directly (read-only) for every `Horse` row before writing the migration. Found exactly one record named `Binx` (`sex: GELDING`, `status: ACTIVE`, no notes) — confirmed with James this was the barn cat before the migration's `UPDATE "Animal" SET species = 'CAT' WHERE name = 'Binx'` was written. The other three real records (Remus, Aries, Prada) were left as `HORSE`.
- **Code:** `src/app/horses/` → `src/app/animals/` (directory rename, including `HorseFormFields.tsx` → `AnimalFormFields.tsx`), `src/app/api/horses/[id]/photos/` → `src/app/api/animals/[id]/photos/`, `createHorse`/`updateHorse` → `createAnimal`/`updateAnimal`, `createHorseMetric` → `createAnimalMetric`, `src/lib/prisma.ts`'s `trackedModels` array updated. R2 storage key prefix for photo uploads changed from `horses/<id>/...` to `animals/<id>/...` for new uploads (old photos keep their already-stored full URLs, so nothing broke there).
- **User-visible copy deliberately did not change** — nav links, page headings ("Horses", "Add horse", "Create horse"), and button text still say "Horse" throughout, matching V2.md's own stated convention that staff-facing language stays "horse" even though the schema says `Animal`. Only routes, file paths, and code identifiers moved.
- **`ChangeLog` history was left as-is** — old rows say `entityType: "Horse"`/`"HorseMetric"` since that was accurate when they were written (§4's append-only principle); only new writes use `"Animal"`/`"AnimalMetric"`.
- **Both test suites updated and fully green**: `tests/vitest/actions/horses.test.ts` → `animals.test.ts`, `tests/e2e/horses.spec.ts` → `animals.spec.ts`, plus every other test file/helper touching `horseId`/`createHorse`/`prisma.horse*`. Confirmed via `npm run test:db:reset && npm run test:unit` (81/81 passing) and `npm run test:e2e` (21/21 passing) after the rename — not just typecheck/lint. `npx tsc --noEmit` and `npm run lint` also both pass clean (had to clear a stale `.next/` build cache first, which still referenced pre-rename paths from an earlier `next dev` run).
- **No new feature work rode along with this** — no species picker in the UI, no species column shown anywhere yet. `species` is populated correctly in the schema and migration; surfacing it is left for whenever it's actually needed.
- **Not yet applied to the live Neon database** — this was deliberately held back for a separate, explicit confirmation before running `prisma migrate deploy` against production, since it's a real schema change plus a data backfill on live records. See "What's Next" below.

## Real Bugs Found Tonight (read before touching related code)

This stack runs newer package versions than typical AI training data assumes, and several genuine version-specific traps got caught *before* they became silent failures, by verifying against current docs rather than assuming. Full detail on each is in `CLAUDE.md`; short version here for orientation:

- **Prisma 7** removed `$use` middleware (Client Extensions are the only hook mechanism now — this is what `withChangeLog` relies on), changed the generator provider name and requires an explicit `output` path, requires a driver adapter passed to `PrismaClient`'s constructor, moved connection config to `prisma.config.ts`, and moved the seed command from `package.json` into `prisma.config.ts`'s `migrations.seed` field. Every one of these was a real error message hit and fixed during setup, not a precaution.
- **Neon + Prisma 7** needs two separate connection strings (pooled for the app, direct/unpooled for migrations and the seed script) — collapsing them back into one breaks migrations via PgBouncer's transaction-pooling mode.
- **Next.js 16** renamed `middleware.ts` to `proxy.ts`, made `params`/`searchParams`/`cookies()` all return Promises, and — this one's still an open risk, not fully resolved — May 2026's security patch (fixing a real DoS CVE) introduced a regression where Server Actions using `useActionState` can receive empty `FormData`. Mitigation in place: every form in this app uses a plain `<form action={...}>` without `useActionState`. This worked fine in testing tonight, but it's not a guarantee the underlying issue is fully absent from our exact patch version — if a future form's submitted values come back empty/null unexpectedly, this is the first thing to suspect.
- **Server Actions cap request bodies at 1MB by default**, and the documented workaround (`bodySizeLimit` in `next.config.ts`) has a long, messy history of not reliably working across versions and hosts. This is *why* photo upload was built as a Route Handler with a plain multipart form instead of a Server Action — sidesteps the whole problem, and it's current official guidance, not a workaround of our own invention.
- **Clerk v7**: `auth()` is async and returns `isAuthenticated` explicitly (don't trust bare `userId` truthiness), and `SignedIn`/`SignedOut` don't exist anymore — replaced by a single `<Show when="signed-in">` component.
- **R2 has two different token systems** that both mention R2 permissions, and only one of them actually produces usable S3-compatible credentials. The general Cloudflare API Token builder (My Profile → API Tokens → Custom Token, "Workers R2 Storage" permission group) gives you a token value that is **not** directly usable as an S3 secret key — it needs to be SHA-256 hashed first, which that flow doesn't do for you. The dedicated R2 → Manage R2 API Tokens flow gives you a ready-to-use Access Key ID/Secret Access Key pair directly. This cost real debugging time tonight (`403 AccessDenied` with correct-looking permissions) before the root cause was found.

## Cross-Entity Daily Dashboard — Built This Session

The dashboard called out as item 1 in the prior session's "What's Next": one row per horse, feed/meds/care/metrics/pasture at a glance. No schema changes and no new Server Actions — everything it shows was already being written by the five prior feature slices, this is purely a new read path plus a new page.

- **`src/app/dashboard/page.tsx`**: lists all `ACTIVE` horses (matches the default filter `horses/page.tsx` already uses) with, per horse: current pasture (`PastureAssignment` with `endDate: null`), today's feeding (`FeedingBaseline` rows joined with today's `FeedingOverride` if one was logged, split AM/PM the same way the horse detail page shows it), today's medication status per active `MedicationRegimen` (given/missed/pending, derived from whether today's `MedicationLog` row exists and what it says), open `HealthIssue` rows plus the single most recent `CareEntry`, and the single most recent `WeightEntry`. Read-only — no forms, no writes; logging still happens from each horse's own detail page, this is purely the "what needs attention today" overview.
- **Access:** any signed-in volunteer (`requireVolunteer()`, no role check), matching the existing precedent set by `horses/page.tsx`, `horses/[id]/page.tsx`, and `fields/page.tsx` — none of those role-gate reads today, only writes are role-gated per the Permissions Quick Reference. Kept consistent rather than inventing a new, narrower read boundary for this one page.
- **Query approach:** one query per model across all active horses (`where: { horseId: { in: horseIds } }`), grouped into `Map`s in JS afterward, rather than one query per horse per model — stays a fixed number of round trips regardless of herd size. ## E2E Suite — Actually Run for the First Time (2026-07-18)

The prior session's "run `npm run test:e2e` for real, once" item turned out to hide two real bugs, not just an unexecuted step — the suite as committed could never have passed on any machine.

- **`test:e2e`'s dotenv-cli invocation never loaded `.env`.** `package.json` had `"test:e2e": "dotenv -e .env.test -- playwright test"` — only `.env.test`, which has no Clerk keys. `global-setup.ts` had a comment claiming the real command was `-e .env.test -e .env`, but that second flag was never actually in the script. Result: `clerkSetup()` failed immediately with "You need to set the CLERK_PUBLISHABLE_KEY environment variable," before creating any test users. CI's E2E step was unaffected by this specific bug (the workflow injects the Clerk keys directly via `env:` on that step, bypassing dotenv-cli), but would still fail today on missing repo secrets — see below. **Fixed:** script is now `dotenv -e .env.test -e .env -- playwright test`; confirmed the local test DB URL still wins over `.env`'s real Neon URL (dotenv-cli: first `-e` takes precedence on overlapping keys).
- **The `.test` TLD used for all three E2E test emails (`e2e-*@volunteer-ops.test`) is rejected by Clerk's live API.** RFC 2606 reserves `.test` for exactly this purpose, but Clerk's own email-format validator 422s on it (`form_param_format_invalid`) — confirmed by direct API probe. `.example.com` (also RFC 2606-reserved, guaranteed non-deliverable) is accepted. **Fixed:** `tests/e2e/test-users.ts` now uses `e2e-*@volunteer-ops.example.com`. The three real (disposable) Clerk test users have now been created in the live dev instance.
- **First real run result: 19/21 passing**, after also fixing two locator bugs surfaced by actually running the suite for the first time (`care-and-metrics.spec.ts`: a `getByText` match that hit both a list item and a `<select>` option — strict-mode violation — and a substring match on "Open health issue" that was also matching the always-visible "Open health issue**s**" read-only heading; both are test-only fixes, not app bugs, and both now use scoped/exact locators).
- **Both remaining failures root-caused and fixed same day.** Both turned out to be test bugs, not app bugs, and not related to each other despite looking like one flaky "isolation" problem at first:
  - **`fixtures.ts` used a bare `base.beforeEach()` call at module scope to truncate the DB before every test.** Node/ESM only evaluates a module once per process, so that call only ever executed the *first* time some spec file imported `fixtures.ts` within a given Playwright worker — Playwright attaches the hook to whichever file triggers that first import, and every other spec file sharing the same worker afterward runs with **no reset at all**, silently accumulating rows across tests. Confirmed by instrumenting `resetTransactionalData()` and the hook itself with a file-based trace log (console output isn't reliable here — Playwright's reporter suppresses stdout for passing tests): the hook fired for exactly one file per worker and never fired for the rest. This is a known category of Playwright gotcha — hooks declared in a shared helper module aren't reliably re-registered per file the way **fixtures** are (fixtures resolve per-test through Playwright's fixture graph regardless of module caching). **Fixed:** the truncate is now an auto fixture (`resetDb`, `{ auto: true }`) in `fixtures.ts`'s `test.extend()` instead of a `beforeEach` call. This was the actual cause of `horses.spec.ts`'s "a Shift Lead cannot create a horse" seeing a nonzero horse count — nothing to do with lock contention or connection pools, the DB just was never being cleared for that stretch of the run.
  - **`smoke.spec.ts`'s Admin/Volunteer role-check test requested both `adminPage` and `volunteerPage` fixtures in the same test.** Both depend on Playwright's single per-test `page` fixture (cached once per test), so the second fixture tried to sign in on the *same already-authenticated page* — hence `Clerk: Failed to sign in: You're already signed in`. Nothing to do with cross-test session bleed; it was self-contained within this one test. `fixtures.ts` already had `openAs(role)` built for exactly this "two roles active at once" case (used correctly elsewhere, e.g. `care-and-metrics.spec.ts`'s medication test) — this test just wasn't using it. **Fixed** by switching to `openAs("volunteer")` for the second actor.
- **Suite is now 21/21 passing**, confirmed on a clean `test:db:reset` + `test:e2e` run. Vitest's 81 tests re-confirmed passing too (no regression from the fixture refactor).
