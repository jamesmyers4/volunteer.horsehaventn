# Horse Haven Volunteer App — V2 Buildout Sessions

## How to use this document

This file is broken into **7 sessions**. Each session is meant to be run as its own Claude Code session. At the end of every session:

1. The feature described in that session must be fully implemented.
2. A matching Playwright/TypeScript test suite must be written or extended to cover the new area, following the existing test conventions already in this repo.
3. Claude Code must **STOP** and wait for the user to start a new session before moving to the next one. Do not chain sessions together automatically, even if this file is loaded in full.

Before starting **any** session, read the current `prisma/schema.prisma`, `CONTEXT.md`, and `CLAUDE.md` in the repo. This document describes intent and field-level requirements, not literal Prisma syntax — resolve naming to match whatever conventions those files already establish (cuid IDs, no hard deletes on business entities, camelCase, lookup-table pattern for extensible categories, append-only ChangeLog pattern, baseline+override pattern used for feeding). If a described model conflicts with something that already exists, extend the existing model rather than duplicating it, and note the deviation in your summary at the end of the session.

At the end of each session, create a brief summary and status (Session ## complete) so the human and the agent knows where claude code left off at.

Terminology note: the schema's core entity is `Animal` (generalized from `Horse` to also cover mules, donkeys, minis, ponies, and non-equine animals). This doc uses "horse" in UI-facing language since that's what staff call it day to day, but all schema references mean the `Animal` model.

Roles already established: `ADMIN`, `SHIFT_LEAD`, `VOLUNTEER`, `GUEST`.

## Session 1 — Location Model Generalization (Barn/Stall/Field/Sick Bay/Arena)

**Goal:** Replace/extend the current field-code-only turnout assignment with a generic location model that covers day turnout fields today and barn stalls, sick bay, and the covered arena as they come online, plus a history of where each animal has been.

**Data model:**

- `Location`: id, `type` enum (`FIELD`, `BARN_STALL`, `SICK_BAY`, `ARENA`, `OTHER`), `name`/label (display string), `fieldCode` (nullable, used when type = FIELD, e.g. "L1", "L2" — migrate existing field codes into this), `barnNumber` (nullable int, used when type = BARN_STALL, supports multiple barns later), `stallNumber` (nullable int, used when type = BARN_STALL, range roughly 1–40 in the main barn today), `isActive` boolean.
- `AnimalLocationAssignment`: append-only, id, `animalId`, `locationId`, `period` enum (`DAY`, `NIGHT`), `effectiveAt` timestamp, `recordedById` (volunteer who logged it), `notes` (nullable text). Never update or delete rows here — a move is always a new row.
- "Current location" for an animal/period is derived as the most recent `AnimalLocationAssignment` row for that `animalId` + `period` combo. Do not store a separate "current location" pointer field that could drift out of sync — always derive it.

**Business rules:**

- Full move history exists and is queryable per animal, but it is **not** shown by default anywhere in the UI — surface it behind a "view history" expand/link on the animal's detail page, not on the main dashboards.
- Both day (turnout field) and night (barn/stall, eventually sick bay) locations are tracked independently — an animal can have a current DAY assignment and a separate current NIGHT assignment at the same time.

**Permissions:** Only `ADMIN` and `SHIFT_LEAD` can create new `AnimalLocationAssignment` records.

**Test coverage to add:**

- CRUD + validation for `Location` (correct fields required per `type`, e.g. a `FIELD` type shouldn't accept a `stallNumber`).
- `AnimalLocationAssignment` creation is append-only — verify no update/delete path exists.
- Current-location derivation returns the correct latest row per animal per period, including when DAY and NIGHT differ.
- Permission checks: non-SHIFT_LEAD/ADMIN volunteers cannot create assignments.
- History view returns full chronological list for an animal.

## Session 2 — Training Tier Progression & Compliance Training

**Goal:** Formalize the Green → Orange → Yellow → Blue handler progression with tenure-based eligibility and a human release gate for Blue, plus a separate annual compliance training system for all volunteers.

**Data model:**

- `TierThreshold`: `tier` (GREEN/ORANGE/YELLOW/BLUE, reusing the existing `HandlingColor` enum), `minDaysTenure` (int), `requiresManualRelease` (boolean). Seed: GREEN = 0 days / no manual release. ORANGE ≈ 180 days / no manual release. YELLOW ≈ 365 days / no manual release. BLUE ≈ 730 days / **requires manual release = true**. Make this table admin-editable data, not hardcoded constants, since exact thresholds may get tuned.
- `Volunteer.firstShiftDate` (nullable datetime): set automatically from the volunteer's **first recorded shift/check-in**, not their account creation date. The tenure clock does not start until this is populated. Until it's set, tier eligibility calculations treat tenure as zero.
- `Volunteer.blueReleasedAt` / `Volunteer.blueReleasedById` (nullable): set only when an ADMIN/SHIFT_LEAD explicitly releases the volunteer for Blue. This is independent of tenure — a volunteer can be tenure-eligible for years and never get released if a human decides they're not ready or they've chosen not to take the class.

**Business rules — tier computation (compute at query time, don't store a cached "current tier" that can drift):**

- `computedEligibleTier` = the highest tier whose `minDaysTenure` is met by `(today - firstShiftDate)`, for tiers where `requiresManualRelease` is false.
- `actualTier` = `computedEligibleTier`, **except** it can never reach BLUE unless `blueReleasedAt` is set — a tenure-eligible volunteer without release stays capped at YELLOW for display/access purposes.
- New volunteers default to GREEN.

**Compliance training (separate from tier progression):**

- `TrainingRequirement`: id, `title`, `description`, `fileUrl` (R2-hosted PDF), `isRequired` (boolean), `renewalPeriodDays` (nullable int — e.g. 365 for annual; null means one-time, never expires), `appliesToTier` (nullable — leave room for future tier-specific or Go-Team-level requirements even though none exist yet beyond the general annual one).
- `VolunteerTrainingCompletion`: `volunteerId`, `trainingRequirementId`, `completedAt`, `expiresAt` (computed from `completedAt` + `renewalPeriodDays` if set), self-attestation only for MVP (a timestamped "I read this" acknowledgment — no quiz).

**Test coverage to add:**

- Tier computation: correct tier at exact threshold boundaries, before `firstShiftDate` is set, and the BLUE-capped-without-release case specifically.
- `firstShiftDate` gets populated correctly on first check-in and never overwritten after.
- Manual Blue release: only ADMIN/SHIFT_LEAD can release; releasing a volunteer who hasn't hit tenure yet should be blocked or explicitly flagged (confirm intended behavior with the user if genuinely ambiguous — don't assume).
- `TrainingRequirement` + completion CRUD, expiration calculation, and a query for "volunteers with expired/missing required training."

## Session 3 — Volunteer Tags & Go Team

**Goal:** A generic, extensible tagging system for volunteers, with Go Team as the first real tag, plus a simple admin-facing "eligible soon" report (not an automated gate).

**Data model:**

- `VolunteerTag`: id, `name`, `description`. Seed with "Go Team".
- `VolunteerTagAssignment`: `volunteerId`, `tagId`, `assignedById`, `assignedAt`.

**Business rules:**

- Tags are always manually assigned/removed by ADMIN/SHIFT_LEAD — no automatic tagging.
- Build an "eligibility candidates" report/query: volunteers with `actualTier = BLUE` (i.e., Blue-released, per Session 2) plus some minimum additional tenure since release. Since the exact Go Team tenure requirement isn't settled yet, make this threshold a simple admin-editable config value rather than hardcoding a number — surface it as a report/list for a human to review, not an automatic tag assignment.

**Test coverage to add:**

- Tag CRUD and assignment/removal, permission checks.
- Eligibility candidates query returns the correct set based on tier + tenure-since-release.
- Confirm a volunteer can hold multiple tags (schema shouldn't assume Go Team is the only one that will ever exist).

## Session 4 — Event Scheduler & Signup

**Goal:** Replace the manual email-tracking process with self-service event signup, scoped notifications, and gated visibility for restricted events like the Blue Handler Class.

**Data model:**

- `EventCategory` (lookup, admin-editable): seed with Fall Festival, Outreach/Tabling, Meetup, Training Class, General Volunteer Shift, Other.
- `Event`: id, `title`, `description`, `categoryId`, `startAt`, `endAt`, `locationText` (free text — events can be off-site, e.g. a booth at a town event), `capacity` (nullable = unlimited), `createdById`, `canceledAt` (nullable), `requiredTag` and/or `requiredTier` (nullable — used for gating, e.g. the Blue Handler Class requires `actualTier = BLUE`), `suppressSignupNotifications` (boolean, default false).
- `EventWatcher`: `eventId`, `volunteerId` — additional people who get organizer-level notifications beyond the creator, added at event creation time.
- `EventSignup`: `eventId`, `volunteerId`, `status` enum (`CONFIRMED`, `WAITLISTED`, `CANCELLED`), `signedUpAt`, `canceledAt`.
- `Volunteer.canScheduleEvents` (boolean): independent of role. This is the permission that grants event-creation ability — grant it to Lori and anyone else who needs it without making them a full ADMIN.

**Business rules:**

- Only volunteers with `canScheduleEvents = true` (or ADMIN) can create events.
- A gated event (e.g. Blue Handler Class) is invisible in listings and blocked for signup to any volunteer who doesn't meet `requiredTag`/`requiredTier` — it should not just be hidden, it should also reject a direct signup attempt server-side.
- Signup respects `capacity`: once full, new signups go to `WAITLISTED`. On a `CONFIRMED` cancellation, automatically promote the earliest `WAITLISTED` signup to `CONFIRMED`.
- Notifications: on signup, cancellation, and waitlist promotion, email the organizer (`createdById`) and all `EventWatcher`s, unless `suppressSignupNotifications` is true. The volunteer's own confirmation email to themselves is separate and always sends regardless of that flag (it's their own signup confirmation, not an admin notification).

**Test coverage to add:**

- Event CRUD, `canScheduleEvents` permission enforcement.
- Gated event: hidden from ineligible volunteers in listings, and signup attempts are rejected server-side even if attempted directly.
- Capacity + waitlist promotion logic, including promotion ordering (earliest waitlisted first).
- Notification suppression flag behavior, and that the volunteer's own confirmation still sends when organizer notifications are suppressed.
- Watcher list receives notifications alongside the organizer.

## Session 5 — Shift Templates, Seasonal Hours & Sign-In/Out

**Goal:** Give volunteers a reference for when a shift typically starts/ends, let shift leads adjust their own shift's actual times, and add a kiosk/badge check-in path alongside the existing phone-based flow.

**Data model:**

- `FarmSettings` (singleton row): `activeSeason` enum (`STANDARD`, `WINTER`), admin-togglable. This is a manual switch (farm staff decide when to flip it based on daylight/weather), not date-driven automation — confirm this assumption with the user if it turns out to matter.
- `ShiftTemplate`: id, `name` (e.g. "AM Shift", "PM Shift"), `standardStartTime`, `standardEndTime`, `winterStartTime` (nullable), `winterEndTime` (nullable). Seed roughly: AM Shift 9:00–11:00 standard, PM Shift 4:00–7:00 standard, with winter variants shifted ~1 hour per the user's note (confirm exact winter offsets with the user rather than guessing precisely).
- Per-occurrence override, using the same baseline+override pattern already established for feeding: whatever the existing shift/check-in record is in the current schema, add optional `actualStartTime`/`actualEndTime` fields that a `SHIFT_LEAD` can set for their own shift instance, defaulting to the `ShiftTemplate`'s time for the currently active season if not overridden. `ADMIN` can override a shift lead's override.
- `Volunteer.checkInCode` (unique string, generated at creation): a single code that serves both a kiosk barcode scan and a personal-phone QR flow — same underlying value, different presentation (printed/badge for kiosk, QR-encoded for the volunteer's own device).

**Business rules:**

- Logging hours after the fact pre-fills from the applicable `ShiftTemplate` (adjusted for `activeSeason`) but remains editable — this is a reference, not an enforced punch-clock.
- Kiosk flow is additive, not a replacement for the existing QR/phone flow from the earlier design. A shared tablet at the barn scans a volunteer's `checkInCode` to toggle check-in/check-out with no login required on the shared device itself.

**Test coverage to add:**

- `ShiftTemplate` CRUD, seasonal time resolution (standard vs winter) based on `FarmSettings.activeSeason`.
- Shift-lead override behavior and admin-overrides-shift-lead precedence.
- `checkInCode` uniqueness and lookup correctness for both kiosk and QR paths.
- Kiosk check-in/out toggle logic (no duplicate open sessions, correct state transitions).
- Regression coverage confirming the existing personal-device QR flow still works unchanged.

## Session 6 — Dashboards: Feed Board & Turnout Board

**Goal:** Two read-only, large-screen displays for the feed areas, built on top of the models from Sessions 1–5.

**Feed Board:**

- Spreadsheet-style grid, one row per animal: standardized headshot photo, name, feed type/amount, hay amount, meds/special instructions (reuse existing feeding baseline+override and care-entry models — don't create parallel ones).
- Headshot standardization: enforce a square crop at upload time (client-side crop step before the image goes to R2), not server-side resizing after the fact.
- Read-only for everyone on the TV display. When viewed by `ADMIN`/`SHIFT_LEAD` at a desktop breakpoint, show inline edit affordances for feed/hay amounts and special instructions — this is a breakpoint/role condition on the same view, not a separate route.
- Each row links out to that animal's current location on the Turnout Board.

**Turnout Board:**

- Grouped by `Location` (FIELD type for day view), vertical ordered list per field reflecting herd hierarchy (lead animal at top) — this ordering decision was already made in an earlier session, carry it forward.
- Day/night toggle pulling from `AnimalLocationAssignment` periods (Session 1).
- Read-only for everyone; same admin-desktop-breakpoint edit affordance pattern as the Feed Board for correcting a location assignment on the spot.
- Responsive behavior: TV/desktop shows expanded by default; mobile uses tap-to-reveal in portrait, no forced landscape (matches the pattern already decided for this board).

**Test coverage to add:**

- Feed Board renders correct current feed/hay/instructions data per animal, including override precedence.
- Headshot crop produces consistent dimensions across different source image sizes.
- Turnout Board groups correctly by field and respects herd-hierarchy ordering.
- Day/night toggle reflects the correct `AnimalLocationAssignment` period.
- Feed Board → Turnout Board link navigates to the correct animal/location.
- Edit affordances appear only for ADMIN/SHIFT_LEAD at desktop breakpoint, not on the TV/mobile read-only views.
