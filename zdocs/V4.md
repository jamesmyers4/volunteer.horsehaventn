Session 1 — Kiosk View-Only Role & Role-Based Landing Routes
Goal: Let two always-on TV terminals stay logged in indefinitely under one shared, genuinely read-only account, and stop routing everyone through the same flat link list after login — land each role somewhere useful for what they actually do first.

Data model:

Add KIOSK to the existing Role enum, alongside ADMIN/SHIFT_LEAD/VOLUNTEER/GUEST. Recommended over two alternatives, both rejected: (a) a boolean flag like Volunteer.isKioskOnly, rejected because every future write-capable action would need to individually remember to check it — fragile over time, the same reasoning ChangeLog's auto-capture wrapper already exists to avoid; (b) reusing the existing GUEST role, rejected because GUEST is scoped to actual guest people (vets, visitors) with time-boxed access (accessValidFrom/accessValidUntil) — folding a permanent system display account into that would muddy both uses. Confirm the KIOSK name itself with the user if a different label is preferred; it's not load-bearing.
No other schema changes. A KIOSK-role Volunteer record is created the same way an ADMIN one is — an existing admin manually sets the role via the volunteer management screen. No new signup flow, matching the existing "no self-service promotion path, ever" rule from CONTEXT.md §5.

Business rules:

KIOSK must fail every write-capable permission check application-wide, not just the ones this session touches directly. Audit every existing permission-check function in the codebase for allowlist vs. denylist logic — an allowlist (role === ADMIN || role === SHIFT_LEAD) is safe by default against a new role; a denylist (role !== GUEST) is not, and needs to be found and fixed if any exist. Flag anything found in the session summary rather than assuming the codebase is uniformly allowlist-based.
KIOSK sessions get a stripped-down layout: no admin nav, no check-in nav, nothing rendered except the board content and the link between Feed Board and Turnout Board (Session 2). This is as much a security-by-obscurity measure as a UX one — a TV visible to anyone walking by shouldn't display links to pages it can't actually use anyway.
Role-based landing route after sign-in: KIOSK → Feed Board. VOLUNTEER and SHIFT_LEAD → the check-in page (SHIFT_LEAD's roster/report links live prominently within that page rather than needing a separate landing route — confirm this reads as sufficient once built, rather than assuming it does). ADMIN → the existing admin console. This changes only the default landing route — it does not restrict which pages a role can otherwise reach; existing permission checks on each page remain the actual authorization boundary.
Clerk supports multiple concurrent sessions per account across devices by default, so logging the same KIOSK account in on both TVs should just work — not a task for this session, just don't design around single-session-per-account assumptions if any exist in the current Clerk configuration.

Test coverage to add:

A KIOSK-role account is rejected by every existing write-capable Server Action, not just newly-audited ones — enumerate the actions this touches in the summary.
KIOSK-role layout renders no admin/check-in nav elements.
Sign-in redirect resolves correctly for each of the four roles.
A KIOSK-role account attempting to navigate directly to a restricted URL (not via nav, e.g. typing it) is still blocked by the underlying permission check, not just hidden from nav.

Session 2 — Feed Board / Turnout Board Kiosk Experience
Goal: Make the Feed Board the default kiosk view with automatic AM/PM switching at noon, a clear way to reach the Turnout Board and back without losing the kiosk's read-only guarantee, and a refresh cadence appropriate for a screen that's never closed.

Before implementing: read the current Feed Board implementation. Its AM/PM behavior (manual toggle, automatic, or absent) is unconfirmed — this session's scope depends on what's actually there today. State what you found in the session summary before describing what you changed.

Business rules:

Feed Board defaults to AM data before noon, PM data at or after noon (local time), re-evaluated on each refresh cycle, not just on initial page load — a screen left open across the noon boundary must flip on its own, not require a reload. Keep a manual override toggle available to any viewer (kiosk or logged-in) who wants to check the other shift's chart without waiting — this is a display toggle, not a write action, so it's fine even under KIOSK's read-only restriction.
Add a clearly visible link from Feed Board to Turnout Board. Modal-over-the-page or full-page navigation-with-a-back-path are both acceptable — match whichever pattern the app already leans on elsewhere rather than introducing a third. Either way, returning from Turnout Board must land back on Feed Board with its current AM/PM state intact, not reset to a default.
Turnout Board must enforce the same KIOSK read-only treatment as Feed Board — no edit affordances rendered for KIOSK regardless of screen size/breakpoint, even though the existing desktop-breakpoint edit affordance for ADMIN/SHIFT_LEAD (from V2 Session 6) still applies to those roles on their own devices.
Both boards poll for fresh data on an interval — every 30 minutes is the target; confirm this already exists from V2 Session 6 and only add it if it doesn't. An always-on screen should never require a manual reload to reflect a same-day change.

Test coverage to add:

AM/PM auto-switch resolves correctly on both sides of the noon boundary, and re-evaluates on a refresh tick rather than only at initial load.
Manual override toggle works independently of the automatic switch and is available under a KIOSK session.
Turnout Board renders zero edit affordances for a KIOSK-role viewer at any breakpoint.
Navigating Feed Board → Turnout Board → back preserves the Feed Board's current AM/PM state.
Refresh interval actually re-fetches data (not just re-renders stale state) at the configured cadence.
